## 已鎖定規格（請勿改動）

**場地**：6m（長軸，兩球門連線方向）× 3m（寬）× 3m（高），封閉空間，六面邊界。
中線在長軸 3m 處，分兩半場各 3m 深。

**隊伍與站位**：3v3，紅隊一端、藍隊另一端，站在 3m 寬的窄邊，相距 6m，面對面。

**球門方向（已修正的關鍵認知）**：
- 兩個門環沿長軸排列，從任一隊的窄邊站位往場內看，**同時看得到兩個門環**。
- 近端門（己方那側的端牆）= 要防守的門
- 遠端門（對方那側的端牆）= 要攻向的門
- **不存在「球門在背後」的情形**
- 設計文件路徑：`docs/drone-soccer-design.md`（§2.1、§2.5 已依此修正）

**球門規格**：門中心高 1.5m，內徑約 1.2m（可微調）。

**視角**：窄邊定點視角，攝影機高度 1.5m（與球門中心等高），水平朝場中央/對方看。
紅藍視角相反。可用 C 鍵切 FPV。

**賽制**：1 局 3 分鐘，進球多者勝（課堂簡版）。

**碰撞**：A 版（玩家互不碰撞）先行。

**前鋒**：每隊 1 名，固定 / 隊伍自選，整局不變，老師指定或學生自選（介面待定）。

**計分**：只有前鋒朝正向完整穿過對方球門（遠端門）才得分。得分後全隊退回自家半場過中線。

**分隊**：伺服器預設自動平均分，老師後台可手動調整。

## 待確認的設計點（§2.1 待確認方塊列出）

1. **得分方向偵測**：計分只看前鋒穿越遠端門，近端門穿越事件要不要偵測（顯示「防守被破」提示）？
2. **門中心高 1.5m 與視角等高**：視線穿過兩個門的圓心，確認這是預期效果。
3. **半場/中線**：中線位在 6m 長軸正中（距兩端門各 3m），退半場 = 越過中線，不是退到己方近端門後面。

**Why:** 使用者在 2026-06-22 指出文件 §2.1/§2.5 的「球門在背後」描述錯誤，並說明正確的長軸排列設計。
**How to apply:** 未來討論計分偵測、攝影機、出生點或半場重置時，務必以「近端=防守門、遠端=攻擊門」為基準，不得使用「背後的門」說法。
